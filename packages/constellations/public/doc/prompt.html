Regeneration Prompt

A complete English specification of Constellations — sufficient to rebuild the app from scratch.  ·  ← Back to app

This document describes every feature of Constellations without prescribing layout or visual design. It is intended as a prompt for an AI coding agent to regenerate the application.

1 Core Concept

Build an interactive knowledge graph explorer called Constellations. The user types any person, place, work, or concept and the app expands it into a network of related entities, which can themselves be expanded. The graph grows outward from the seed term, revealing connections between people, events, works, and ideas.

The organizing principle is a strict bipartite graph: the graph alternates between two kinds of nodes — atomic entities (fundamental individuals such as people, ingredients, or symptoms) and composite entities (collections or works such as events, recipes, or diseases). Edges only connect an atomic node to a composite node — never two nodes of the same kind. This constraint gives the graph its navigable structure.

2 Bipartite Pairs

The system supports multiple bipartite pairs, all using identical mechanics. When the user searches a term, the system classifies it and locks a pair for the entire graph — the pair never changes mid-session.

Classification uses an LLM call with structured JSON output. Fast heuristics skip the LLM: a DOI/arXiv pattern implies Author↔Paper; a parenthetical disambiguator like "(album)" implies Person↔Event. The LLM returns type, is_atomic flag, atomic type label, composite type label, and a brief reasoning string.

3 Data Sources

Gemini (Google AI) — primary intelligence
  • Classify a search term into its bipartite pair and atomic/composite role
  • Given a composite node, return 8–12 connected atomic nodes (e.g. actors for a film)
  • Given an atomic node, return 8–12 connected composite nodes (e.g. events for a person)
  • Given two terms, find a connecting chain of intermediary nodes
  • All calls receive Wikipedia article text as verified context to reduce hallucination
  • Returns structured JSON including entity name, type, year, role, and a one-sentence evidence snippet
Wikipedia
  • Full plain-text article extract (not just intro) used as verified context for Gemini
  • Canonical title resolution with redirect following
  • Page IDs as the strongest deduplication key across nodes
  • When a term has no article, find Wikipedia pages that mention it and use those as context
Wikidata
  • For organization nodes: retrieve founders, directors, CEOs, and key people
  • For film/TV titles: retrieve cast and crew
OpenAlex (academic mode)
  • Look up authors by name, retrieve their most-cited papers
  • Look up papers by title and retrieve co-authors
  • Crossref is the fallback for DOI lookups
Images (server-side)
  • Server proxies image fetches to avoid browser CORS restrictions
  • Primary source: Wikipedia Commons
  • Fallback: DuckDuckGo image search

4 Search Modes

Explore

The user enters a single term. The system classifies it, creates a seed node, then immediately expands it to show 8–12 connected neighbors. The user can continue expanding any node in the graph. URL parameter: ?q=term auto-starts an explore search.

Connect (Pathfinding)

The user enters two terms. The system asks the LLM to find a chain of intermediary nodes connecting them, then renders the entire path as a graph. URL parameters: ?start=term&end=term auto-start a path search.

5 Graph Interaction

Node actions

  • Click — select node, show details in sidebar
  • Double-click — open context menu
  • Expand — fetch and display connected neighbors via AI or cache
  • Expand More — force fresh expansion, passing existing neighbors to avoid duplicates
  • Expand Leaf Nodes — bulk-expand all unexpanded children of the selected node
  • Delete — remove node and all edges connected to it; optionally prune the disconnected subtree
  • Find Better Photo — re-query image sources for the selected node

Edge actions

  • Click edge — show evidence snippet in sidebar (one sentence justifying the connection, with a link to the source)

Canvas

  • Pan and zoom the entire graph
  • Drag individual nodes to reposition them (pins them in place)
  • Arrow keys navigate between nodes in timeline mode

Auto-expand

After expanding a node, if it receives fewer than three neighbors, the app automatically queues a second "expand more" call. This ensures nodes with sparse initial results still build out a useful subgraph. The auto-expand fires at most once per node.

6 Evidence System

Every edge carries an evidence object with: a source kind (wikipedia, openalex, crossref, or ai), a one-sentence snippet justifying the connection, and a URL to verify it. Evidence is extracted from the verified context text used during expansion — not generated separately. Clicking an edge shows this evidence in the sidebar.

7 Caching and Persistence

All expansions are cached in a PostgreSQL database so repeated queries return instantly without calling the AI.

Database schema

  • nodes — title, type, is_atomic, wikipedia_id (unique), description, year, image URL, wiki summary, JSON metadata
  • edges — atomic_id (FK), composite_id (FK), label, JSON metadata (evidence)
  • saved_graphs — name (unique), full graph state as JSON, updated_at

Cache behavior

On expansion, the server is queried for cached neighbors by node database ID. A cache hit returns all previously fetched neighbors immediately. On a miss, AI is called and the result is saved for future sessions. Wikipedia page ID is the strongest deduplication key; title + type is the fallback.

Save / load graphs

Users can save a named snapshot of the current graph (nodes, links, display settings, search context) and reload it later. Graphs can also be exported as JSON files and imported.

8 Display Modes

Force-directed graph (default)

Nodes positioned by a D3.js physics simulation with repulsion, link forces, and collision detection. Nodes are draggable. The canvas is pan/zoomable.

Timeline mode

Only composite (non-atomic) nodes are shown, arranged chronologically on a horizontal axis by their year field. Useful for seeing the arc of a person's career or a historical period.

Text-only mode

Suppresses all image loading. Useful on slow connections.

Compact mode

Reduces node card sizes and overall visual density.

9 Node Display

Atomic nodes (people, etc.) render as circular cards with a photo and name. Composite nodes (events, films, etc.) render as rectangular cards with a title, year, and brief description. Both types show a loading indicator while expanding.

Each node stores: title, type, is_atomic flag, Wikipedia page ID, year, image URL, short description/wiki summary, and a classification reasoning string (shown in admin mode only).

10 People Browser

A panel lets users browse Wikipedia's most-notable people filtered by nationality, occupation, and birth century, using a pre-loaded JSON dataset. Clicking a person starts a search for them.

12 Kiosk / Domain Mode

The app can be locked to a specific subject domain (History, Film, Literature, Music, Science, etc.). The selected domain is passed as context to Wikipedia and Gemini to disambiguate searches. Each domain has a label, description, and a list of seed terms. Admin mode allows editing domains in-app.

13 Chrome Extension

  • A background service worker registers a right-click context menu: "Graph '[selection]' on Constellations"
  • Selecting text on any page and choosing this option opens the side panel and starts a search
  • Clicking the extension icon opens a popup that auto-searches the current page title (stripping suffixes like "— Wikipedia" or "— IMDb")
  • The side panel renders the full app in a compact layout with evidence popups as floating overlays

14 Backend Server

An Express server handles:

  • Node upsert and lookup by title or Wikipedia page ID
  • Expansion cache read (GET) and write (POST)
  • Saved graph CRUD
  • Image fetching proxy (avoids browser CORS)
  • AI proxy: forwards Gemini calls using a server-side API key, so users don't need their own

The server connects to PostgreSQL. All tables have permissive access policies since the data is non-sensitive.
